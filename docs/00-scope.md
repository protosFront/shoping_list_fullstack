# 00. Область проекта (Scope)

## Цель
Сделать полноценное full-stack приложение “Список покупок” для демонстрации инженерных навыков:
Nuxt (frontend) + Fastify (backend) + PostgreSQL (DB) + Docker + деплой на VPS с HTTPS + Swagger.
Проект должен быть простым по домену, но зрелым по реализации (безопасность, миграции, контракт API, качество).

## MVP — что делаем обязательно
- Авторизация: регистрация, логин, логаут, refresh-сессии, эндпоинт `/me`.
- Списки: CRUD (создать, список, переименовать, удалить).
- Товары: CRUD + переключение `куплено/не куплено`, фильтры (все / активные / купленные).
- Шаринг списков:
  - роли: Owner / Editor / Viewer
  - приглашения по email (invites) + принятие приглашения
  - просмотр участников (все добавленные в список) + управление участниками (только владелец)
- PWA (MVP-уровень):
  - установка приложения (installable)
  - cache app shell / статических ресурсов
  - офлайн-синхронизация данных и синхронизация изменений
- Realtime-коллаборация (WS обновления).

## Out of Scope — чего НЕ делаем в MVP
- Публичные ссылки “без аккаунта” (анонимный доступ).
- Категории товаров, сканер штрих-кода, аналитика расходов.
- Платежи, уведомления, интеграции, админка.

## Роли и краткие права
- Owner: полный доступ к списку/товарам + управление участниками/приглашениями.
- Editor: доступ к списку + управление товарами (создать/изменить/удалить).
- Viewer: доступ к списку только на чтение.

## Демонстрационный сценарий (60 секунд)
1) Логин.
2) Создать список “Дом”.
3) Добавить товары: “Молоко”, “Яйца”.
4) Отметить “Молоко” как куплено.
5) Пригласить пользователя `viewer@test.com` как Viewer.
6) Под другим пользователем открыть список и увидеть товары (без возможности правок).
7) Показать PWA: установка (install) и запуск как standalone.

## Нефункциональные требования (минимум)
- Единый формат ошибок API (code/message/details/requestId).
- Валидация входящих данных на API.
- Логи запросов на API.
- Миграции БД (Prisma) и seed для демо-данных.
- Swagger/OpenAPI доступен на `/docs`.
