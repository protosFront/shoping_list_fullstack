# WORKFLOW — как мы работаем по проекту

## Главный принцип
Источник правды — файлы в `docs/`, а не переписка.
Переписка нужна для обсуждения, решения фиксируем в `docs/` и коммитим.

## Роли
- Ты: реализуешь код, принимаешь архитектурные решения.
- Я: аналитик/архитектор/ревьюер — задаю рамки, предлагаю варианты, ловлю риски, формулирую критерии готовности.

## Структура общения
В проекте держим 3 чата:
1) Roadmap (Master) — план, статусы, решения (только финальные итоги)
2) Backend (Fastify) — обсуждение API/DB/Auth/Sharing
3) Frontend (Nuxt) — UX, страницы, PWA, клиент API

## Как начинается любой шаг
Ты пишешь:
- какой milestone/шаг делаем (например: M1.2)
- что уже сделано (1–5 пунктов)
- что неясно (вопросы)
- (если есть) прикладываешь соответствующий кусок docs или кода

Я отвечаю:
- цель шага
- список задач (наводящие пункты)
- что изучить, чтобы сделать
- критерии готовности (acceptance)
- риски/подводные камни

## Как закрываем шаг
Ты присылаешь:
- что сделал
- ссылки/скрины (если надо)
- изменения в docs (текстом или файл)
Я:
- делаю ревью
- говорю “шаг закрыт” и фиксирую следующий

## Правила качества (Definition of Done)
Шаг считается готовым, когда:
- есть артефакт (код/док/скрин/ссылка)
- есть критерии готовности и они выполнены
- нет критических дыр в безопасности/валидации
- запуск повторяем (локально одной командой)

## Как предотвращаем потерю контекста
- `docs/PROJECT_CONTEXT.md` обновляется при крупных решениях.
- В начале нового чата ты вставляешь PROJECT_CONTEXT целиком (если нужно).
